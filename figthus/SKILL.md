# Figthus 正文配图

## 核心定位

Figthus 为中文文章设计和生成 16:9 横版正文配图。目标不是做商业插画、PPT 信息图、教程页或可爱文具海报，而是先理解文章里的认知锚点，再把其中一个判断、流程、结构、状态或隐喻，转成一张清爽、怪诞、有记忆点的手绘解释图。

默认视觉 IP 是一对残损文具：**烂笔头** 和 **破橡皮**。

- 烂笔头负责先写、先错、先把混沌外化成痕迹。
- 破橡皮负责怀疑、擦改、校准、把假清楚擦出空白。

一句话：**不画知识点，画知识点里正在发生的那一下。**

按任务需要读取，不要一次塞满上下文：

## 工作流

### 1. 消化正文

先读用户给的正文、链接、Notion 页面、Markdown 文件或截图内容。提炼：

- 核心判断是什么
- 哪些段落承担认知转折
- 哪些内容适合视觉化
- 哪些地方只适合文字，不需要图
- 哪些地方有摩擦、代价、误区、断点、证据、修正或停止条件

不要平均配图。优先选择“认知锚点”，例如：第一版被推出去、证据被拖回来、假清楚被擦掉、失败被分类、死循环被闸门拦住、好句子从废话里被擦出来、空白被重新开出来。

### 2. 把概念换成可视动作

不要直接画抽象名词，例如“AI Agent”“Loop Engineering”“反馈机制”“知识管理”。先把它们转成能被身体理解的动作：

```text
文章观点 -> 认知摩擦 -> 可视动作 -> 角色分工 -> 低科技隐喻 -> 插图
```

例如：

- “不是多试几次” -> 误区 -> 死循环被拦住 -> 破橡皮守停止闸。
- “结果要带证据回来” -> 证据回流 -> 蓝色证据单被拖回 -> 破橡皮拿证据，烂笔头准备改。
- “先有第一版才知道错在哪” -> 外化 -> 烂笔头推出第一条歪线。

### 3. 先出配图策略

如果用户只是说“分析怎么配图 / 思考哪些地方需要配图 / 先不要生图”，先给 shot list。每张图写清楚：

- 放在哪个段落后
- 图的主题
- 认知锚点
- 核心意思
- 结构类型
- 可视动作
- 烂笔头在做什么
- 破橡皮在做什么
- 角色状态：断芯 / 裂缝 / 裂开了 / 胶带修补
- 建议元素
- 建议中文标注词

默认 4-8 张。文章很短时 1-3 张；长文也不要轻易超过 9 张。够用就好，避免把正文做成画册。

### 4. 单张生成

如果用户明确要求“生成 / 输出 / 做图 / 帮我生成”，不要停下来等确认；用内置 `image_gen` 每张单独生成。不要把多张图拼在一张里。

每张图只讲一个核心结构或认知动作。提示词必须包含：

- 16:9 横版中文正文配图
- 纯白背景
- 黑色手绘线稿
- 少量红色/橙色/蓝色中文手写批注
- 大量留白
- 烂笔头与破橡皮的角色分工，或明确说明为什么只出现其中一个
- 画面里留下的痕迹、擦痕、灰屑、断芯或空白
- 禁止 PPT、商业插画、儿童文具海报、复杂架构、左上角类型标题

不要复刻过往案例。角色设定图只用于校准形象，不要复用它的分镜、姿势或标注作为最终正文插图。

### 5. 检查与迭代

生成后检查 `references/qa-checklist.md`。如果出现以下问题，优先重生成或局部编辑：

- 角色像儿童文具吉祥物
- 烂笔头太完整、太新、没有大包圈和残橡皮头
- 破橡皮太完整、太干净、粉色侧没有残损
- 两个角色分工不清
- 画面没有痕迹与空白的关系
- 画面太满
- 太像流程图/PPT/课程页
- 中文太多或错字严重
- 左上角出现“常见坑/流程图/系统架构图/路线图”等标题
- 背景不是干净白底

### 6. 保存交付

如果用户在 workspace 内工作，把最终图复制到：

```text
assets/<article-slug>-figthus/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

## 输出口径

生成前的策略输出要短而准。生成后的交付要包含：

- 生成了几张
- 每张图的用途
- 保存路径
- 哪些图最稳，哪些图是可选

不要长篇解释风格理论；让图自己说话。
